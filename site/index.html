op-who

A macOS menu bar utility that shows which process triggered a 1Password approval dialog — whether from the CLI (op) or the SSH agent.

Install Latest release

Why does it exist?

1Password's approval dialog tells you which app is requesting access — but not which terminal session or which command started it. If you have several terminals open running builds, git operations, or Claude Code sessions, you're left guessing.

op-who fills in the missing context so you can approve (or deny) with confidence. When 1Password shows its approval dialog, op-who pops up a floating overlay right next to it, telling you exactly what asked.

What it shows

A floating overlay that appears next to the 1Password dialog and reports:

The full process chain

From the trigger up to the terminal, e.g. op → bash → node → zsh, with verified op binaries shown in green.

Working directory

The CWD of the requesting process, walked up the chain to the first meaningful directory.

Claude Code context

Session/project name and the last prompt or command, when Claude Code is in the chain.

Terminal tab

The terminal tab title matched by TTY, plus a button to jump straight to that tab.

PID & TTY

The process ID, the TTY device, and how long the approval has been pending.

A readable summary

One plain-English line — "Claude Code session 'op-who' is signing a commit" — from a configurable rule engine.

How it works

  1. Watches for the 1Password process and verifies its code signature (Apple Team ID 2BUA8C4S2C).
  2. Attaches an Accessibility observer to detect new windows, and classifies approval dialogs by their bare 1Password title.
  3. Finds trigger processes: op CLI processes (signature-verified) or SSH clients (ssh, git, scp, sftp, rsync, ssh-keygen, op-ssh-sign).
  4. Walks each trigger's parent chain, stopping at macOS app boundaries (1Password already shows the app name).
  5. Looks up terminal tab titles via AppleScript (Terminal.app, iTerm2) or the Accessibility API (Ghostty, Warp, and others).
  6. Runs the captured context through a user-editable rule engine to produce the one-line description.
  7. Shows the overlay near the dialog, then dismisses it automatically when the dialog closes or the trigger processes exit.

Install

op-who is a notarized, Developer ID–signed macOS app. Install it with Homebrew:

brew install --cask stigsb/tap/op-who

brew upgrade --cask op-who keeps it current.

Prefer a manual install? Download op-who.zip from the latest release, unzip it, and drag op-who.app to /Applications. For MDM/Fleet deployment, each release also ships a signed op-who-<version>.pkg installer.

After install, look for the op-who icon in your menu bar and grant Accessibility once when prompted (System Settings → Privacy & Security → Accessibility). When 1Password shows an approval dialog, the overlay appears next to it.

Security

op-who validates the identity of the processes it interacts with.

  • 1Password app — code signature verified before attaching the Accessibility observer.
  • op CLI — executable path resolved and code signature checked; verified binaries shown in green, unverified in orange.
  • TTY paths — validated against /dev/ttys[0-9]+ before any read/write operation.
  • TTY messages — only written when you explicitly click "Send Message" in the overlay.

It runs as a menu bar app (LSUIElement) and is non-sandboxed because it must enumerate other processes and read their working directories. It never reads secret values — only process metadata. Release-artifact trust is documented in SIGNING.md.

Configurable rules

The overlay's one-line description comes from a rule engine. A default set of built-ins covers common cases (git push, op read, SSH key signing, Claude plugin update checks); you can disable any built-in or add your own from Settings using NSPredicate expressions.
